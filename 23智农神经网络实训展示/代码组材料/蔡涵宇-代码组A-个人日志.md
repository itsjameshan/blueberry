## 第二天（周二）—— 模型分析与系统联调

###  当日工作内容

####  早上：训练结果分析与模型评估

- **Stand-up 汇报**：汇报昨日环境部署、夜间模型训练进度及现存阻塞问题，完整展示 YOLOv11 与 YOLOv12 两组训练结果。
- **核心指标解读**：逐项解读 `results.csv` 中的精确率（Precision）、召回率（Recall）、F1 分数、mAP 等核心指标，深入认识模型评估需要结合错检、漏检和实际检测效果综合判断，而非只看单一数值。
- **推理链路拆解**：拆解模型完整推理流程，讲解图像预处理、后处理、NMS 非极大值抑制、置信度阈值调节原理；完成多组阈值对比测试并记录数据。
- **双模型对比**：制作 **YOLOv11 vs YOLOv12 训练指标对比表**，分析两类模型在检测速度、小目标识别精度上的差异；出具模型选择建议文档，将训练可视化图表与预测结果图移交发布组 A 用于 PPT 制作。

####  下午：ONNX 模型接入与接口联调

- **模型接入后端**：将训练生成的 `v11best.onnx`、`v12best.onnx` 模型接入 Flask 后端 `app.py`，联调**单图检测、批量图片检测、大图裁剪重建、检测结果存储与下载**全套接口。
- **接口 Bug 排查**：逐一排查文件路径错误、ONNX 加载失败、数据库字段不匹配、前端图片渲染异常等问题，整理《接口联调问题清单》，逐一修复验证。
- **跨组技术支持**：持续派驻组员为其余小组提供代码与模型技术答疑；帮助科研组核对训练结果图，协助发布组理解检测截图与模型评估图表。

##  实训整体总结
**代码理解 → 环境部署 → 数据标准化 → 模型训练 → ONNX 部署 → 后端联调 → 系统演示 → 成果归档** 的全流程闭环。我们深刻认识到：

1. **数据标注质量决定模型精度上限**——标注规范必须在前端统一，标注完成后需交叉检查
2. **模型指标必须结合实际场景解释**——单一数值无法反映真实检测效果，需结合错检漏检案例综合判断
3. **工程交付不只是"能跑"**——稳定性、文档完整性、展示可复现性同样重要
4. **跨组沟通决定项目效率**——技术方案需要用通俗语言向非代码成员讲解，同步需求需前置
